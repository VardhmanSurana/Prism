# Plan: NLE Video Editor Bug Fixes

## Summary

5 bugs/issues in the Prism NLE (video editor). Ordered by impact ‚Äî audio mixer is P0, remainder are P1/P2.

---

## P0 ‚Äî Audio mixer is completely non-functional

**Bug:** `useAudioMixer` is called in `VideoEditorMode` with `videoRefsState` (a `Map<id, React.RefObject>`) built via `React.createRef()` on every render cycle. Those refs are **never attached to any `<video>` DOM element** ‚Äî the actual video elements live inside `PreviewArea` (`videoRef`, `transitionVideoRef`, `additionalVideoRefs`). As a result `videoEntry.current` is always `null`, so `createMediaElementSource` never runs, gain nodes are never created, and multi-track audio/fade automations are dead. Only the primary clip's audio plays via native `<video>` playback.

**Affected files:**
- `frontend/hooks/useAudioMixer.ts`
- `frontend/components/VideoEditing/VideoEditorMode.tsx`
- `frontend/components/VideoEditing/VideoEditorMode.tsx` (PreviewArea)

**Recommended solution (Option A ‚Äî recommended, minimal refactor):**
1. Move `useAudioMixer` call from `VideoEditorMode` into `PreviewArea`.
2. Change `useAudioMixer`'s first parameter from `Map<string, React.RefObject<HTMLVideoElement | null>>` to `Map<string, HTMLVideoElement | null>` (PreviewArea already stores elements directly in `additionalVideoRefs`).
3. In `PreviewArea`, build a `clipElementsRef = useRef<Map<string, HTMLVideoElement>>(new Map())`.
4. Maintain it:
   - Primary slot: key it by `activeClip.id`; when `activeClip` changes, swap the map key (remove old, insert new). The actual `videoRef.current` is reused.
   - Transition slot: key by `transitionState.nextClip.id`.
   - Additional slots: already keyed by `clip.id` via the ref callback on each `<video>`.
5. Pass `clipElementsRef.current`, `tracks`, `isPlaying`, `playheadPosition`, `projectFps` to `useAudioMixer`.
6. Clean up the dead `videoRefsMap` / `videoRefsState` plumbing from `VideoEditorMode`.

**Rationale:** `PreviewArea` already owns every `<video>` element that exists in the DOM. Keeping the mixer close to its data source removes the indirection (and dead code) without changing any external API.

## P0 ‚Äî Incorrect solo logic in audio mixer

**Bug:** In `useAudioMixer.ts:69-71`, when a track is soloed, the per-clip `effectiveVolume` is set to `clipVolume` for **every clip in every track** ‚Äî solo doesn't actually isolate the soloed track; it just unmutes everything that isn't clip-muted.

**Affected files:** `frontend/hooks/useAudioMixer.ts`

**Solution:** Compute `anyTrackSoloed = tracks.some(t => t.solo)`. Then for each clip:
- `if (anyTrackSoloed)` ‚Üí `effectiveVolume = (track.solo && !clip.muted) ? clipVolume : 0`
- `else` ‚Üí `effectiveVolume = (trackMuted || clipMuted) ? 0 : clipVolume`
Keep the WebAudio `gain.setValueAtTime` call.

## P1 ‚Äî Keyframes lost when a clip is split

**Bug:** In `nleStore.splitClip` (`store/nleStore.ts:451-458`), `clip2.keyframes = {}`. All keyframe animation on the trailing half is silently discarded.

**Affected files:** `frontend/store/nleStore.ts`, `frontend/lib/keyframes.ts`

**Solution:**
1. Add a helper `splitKeyframes(keyframes, splitTime)` in `frontend/lib/keyframes.ts` that returns `{ before: Keyframe[], after: Keyframe[] }`.
2. In `splitClip`:
   - `clip1.keyframes = splitKeyframes(clip.keyframes, splitRelativeFrame).before`
   - `clip2.keyframes = shiftKeyframes(splitKeyframes(clip.keyframes, splitRelativeFrame).after, -splitRelativeFrame)`
3. `shiftKeyframes` subtracts `delta` from every keyframe's `t`.

## P1 ‚Äî MLT transitions don't render in export

**Bug:** In `nle_engine.py:547-553`, all transition elements use `a_track="0"` and `b_track="0"` (same index). MLT requires transitions to span two distinct multitrack playlist entries.

**Affected files:** `backend/app/services/nle_engine.py`

**Solution options:**

**Option A (minimal, safe for now):** In `_build_tractor`, skip emitting `<transition>` elements for transitions within the same playlist. Add a `TODO: same-track transitions require split playlists` comment. Exported video renders a hard cut; preview remains accurate.

**Option B (correct):** Restructure the first video track's MLT entry so that each consecutive clip-pair with a transition is placed on its own pair of playlists (clip A on playlist X, clip B on playlist Y) for the overlap duration only, with the rest of the timeline on separate playlists. This is a significant rewrite of `MLTBuilder._build_tractor`/`_create_producer`.

**Recommended:** Option A now, backend TODO for Option B later. Same-track transition rendering in MLT is non-trivial; a hard-cut export with live preview transitions is a better user experience than a broken render.

## P1 ‚Äî Stale undo/redo button `disabled` state

**Bug:** In `VideoEditorMode.tsx:386,397`, `disabled={!useNLEStore.getState().canUndo()}` reads store state via `getState()` outside React's render cycle, so the disabled attribute is stale and never updates reactively as history changes.

**Affected files:** `frontend/components/VideoEditing/VideoEditorMode.tsx`

**Solution:** Replace with reactive selectors:
```
const canUndo = useNLEStore(s => s.canUndo());
const canRedo = useNLEStore(s => s.canRedo());
```
Drop the `getState().canUndo()` calls from the JSX.

## P1 ‚Äî Wipe transitions render as plain cut in preview

**Bug:** `PreviewArea`'s `transitionState` and `renderWebGL` only handle `crossfade`, `dissolve`, `slide-left`, `slide-right`. Wipes (`wipe-left`, `wipe-right`) fall through to the `else` branch and render as a plain cut.

**Affected files:** `frontend/components/VideoEditing/VideoEditorMode.tsx` (PreviewArea)

**Solution:** For `wipe-left`/`wipe-right`, implement a horizontal translate composite during the transition window:
- Clip A translates in from the opposite direction of the wipe
- Clip B translates in from the wipe direction
- Mirror the existing `slide-left`/`slide-right` logic with swapped direction.

## P2 ‚Äî Move silently no-ops when clip is dropped on another clip

**Bug:** `moveClip` in `nleStore.ts:402-404` returns `{}` (no-op) when the snapped position would overlap an existing clip. The clip snaps back to its original position with zero user feedback, which is confusing.

**Affected files:** `frontend/store/nleStore.ts`

**Solution:** Easiest fix is fine and doesn't require UX redesign ‚Äî return the clip to its original start frame when overlap is detected (already implied by the no-op, but the re-render shouldn't teleport). Currently, because `set((s) => ({ ... }))` returns `{}`, Zustand doesn't update, but the on-screen position might still show the dragged clip briefly. To fix it cleanly, emit the original state on overlap (a no-op state change is actually fine ‚Äî Zustand set with `{}` won't re-render, so the visual is just the last valid position). Add a small audible/visual snap or tooltip "Can't drop here" if you want feedback (out of scope for now). Functional fix: document and accept. The current behavior is actually safe ‚Äî just confusing. Reserve this for a P2 polish pass.
