# Prism NLE Video Editor — Implementation Plan

## Goal

Add a full Non-Linear Editor to Prism — multi-track timeline, clip trimming/splitting, keyframeable effects, text overlays, transitions, audio editing, speed ramps, and non-destructive export. Entry point: click "Edit" on a video in the Lightbox. Design constraint: **simpler than Premiere, powerful underneath**.

---

## Core Technical Decision: MLT Framework

**Rendering engine: MLT** (the engine behind Shotcut/Kdenlive — LGPL, mature, ships as `melt` CLI + Python bindings).

Why MLT over hand-rolled ffmpeg filter_complex or WebGL compositor:
- **Same engine for preview and export** — eliminates the "preview doesn't match export" drift problem entirely
- Already implements: multi-track compositing, keyframeable filter parameters, speed/time-remapping, transitions, color correction (frei0r/avfilter plugins), audio mixing
- Local, offline, no cloud — matches privacy-first positioning
- Python bindings (SWIG) + CLI `melt` slot cleanly into existing FastAPI backend
- Boring, stable, well-understood dependency (like Pillow, OpenCV, ffmpeg)

**Preview strategy: proxies + on-demand frame rendering**
1. On import into editor, generate low-res proxy (640×360, H.264, low bitrate) in background
2. Timeline/preview scrubs against proxies via MLT
3. Final export re-links to full-resolution sources
4. Cache rendered preview frames for current playhead region

User never sees proxy management — pro technique, hidden complexity.

## Architecture

```
Frontend (React/Tauri)
  ├─ Timeline UI (DOM-based — tracks, clips, keyframe editor, playhead)
  ├─ Preview surface (canvas fed frames from backend via WebSocket)
  ├─ Inspector panel (reuses photo editor AdjustPanel/HslPanel/DetailPanel)
  └─ Talks to backend via REST + WebSocket (frame streaming, live params)

Backend (FastAPI)
  ├─ Edit Project store (JSON = source of truth, SQLite for metadata)
  │    New tables: video_projects, video_tracks, video_clips, video_keyframes
  ├─ Project → MLT XML translator
  ├─ MLT engine (python-mlt or subprocess `melt`)
  │    ├─ Preview: renders frames at proxy resolution, streamed via WebSocket
  │    └─ Export: full-res render to MP4/MOV via melt
  ├─ Proxy generation service (background job, reuses ProcessPoolExecutor)
  └─ Reuses: existing SQLite, processing queue, SSE infrastructure
```

## Edit Project JSON Schema (Source of Truth)

Independent of MLT — makes undo/redo, autosave, future EDL/XML export possible without engine lock-in.

```json
{
  "id": "proj_123",
  "name": "My Edit",
  "fps": 30,
  "resolution": {"w": 1920, "h": 1080},
  "tracks": [
    {
      "id": "v1", "type": "video", "name": "Video 1",
      "clips": [
        {
          "id": "clip_1",
          "sourceId": "photo_9f2",
          "in": 0.0, "out": 4.2,
          "trackStart": 0.0,
          "speed": 1.0,
          "transform": {"scale": 1.0, "x": 0, "y": 0, "rotation": 0},
          "keyframes": {
            "opacity": [{"t": 0, "v": 0}, {"t": 0.5, "v": 1}]
          },
          "effects": [
            {"type": "colorGrade", "lut": "warm_01", "intensity": 0.6}
          ],

        }
      ]
    },
    {
      "id": "a1", "type": "audio", "name": "Audio 1",
      "clips": [
        {
          "id": "aclip_1",
          "sourceId": "photo_9f2",
          "in": 0.0, "out": 4.2,
```

Every clip references existing `Photo`/video records by ID — no duplication of ingestion, thumbnail, or metadata pipeline. The video editor is a **consumer** of the library, not a separate silo.

## SQLite Schema

```python
class VideoProject(Base):
    __tablename__ = "video_projects"
    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(String(255))
    cover_photo_id: Mapped[Optional[int]] = mapped_column(ForeignKey("photos.id"))
    width: Mapped[int] = mapped_column(Integer, default=1920)
    height: Mapped[int] = mapped_column(Integer, default=1080)
    fps: Mapped[int] = mapped_column(Integer, default=30)
    project_json: Mapped[str] = mapped_column(Text)  # Full timeline JSON (source of truth)
    created_at: Mapped[datetime]
    updated_at: Mapped[datetime]

class VideoClip(Base):
    """Source media reference — one per unique source file."""
    __tablename__ = "video_clips"
    id: Mapped[int] = mapped_column(primary_key=True)
    photo_id: Mapped[Optional[int]] = mapped_column(ForeignKey("photos.id"), index=True)
    source_path: Mapped[str] = mapped_column(String(1024))
    duration: Mapped[float]
    width: Mapped[int]
    height: Mapped[int]
    fps: Mapped[Optional[float]]
    codec: Mapped[Optional[str]]
    has_audio: Mapped[bool] = mapped_column(Boolean, default=True)
    proxy_path: Mapped[Optional[str]] = mapped_column(String(1024))  # low-res proxy file
    proxy_status: Mapped[str] = mapped_column(String(20), default="pending")  # pending|generating|ready|failed
    audio_waveform_json: Mapped[Optional[str]] = mapped_column(Text)
```

## Phase Breakdown

### Phase 0 — Foundation Spike (1-2 weeks)
**No UI.** Prove MLT pipeline end-to-end.

- Install MLT (`python-mlt7` or `melt` CLI)
- Hand-write one MLT XML with 2 clips + 1 transition + 1 keyframed filter
- Render via `melt` CLI, confirm output is correct and fast enough
- Build `nle_engine.py`: Project JSON → MLT XML translator (basic: trim, concat, one transition)
- Build `nle_preview.py`: render single frame at proxy resolution
- Build `nle_proxy.py`: generate low-res proxy from source clip via ffmpeg
- Validate: rendering speed, output quality, transition correctness on target hardware
- **This de-risks everything.** Do not proceed to Phase 1 without this green.

**Deliverable:** Script that takes a hand-written project JSON → generates MLT XML → renders via `melt` → outputs MP4. Confirmed working on dev machine.

### Phase 1 — MVP: Linear Cut Editor (4-5 weeks)
Single video track + single audio track, trim/split/ripple, crossfade transitions, text overlays, volume per clip, export.

**Backend:**
- `VideoProject`, `VideoClip` models + migration
- `POST /nle/projects` (create from photo), `GET/PUT /nle/projects/{id}`
- `POST /nle/clips/analyze` (ffprobe + waveform)
- `POST /nle/preview` (single frame via MLT at proxy resolution)
- `WebSocket /nle/preview/stream` (frame streaming for playback)
- `POST /nle/export` + status/download (full-res via MLT)
- Proxy generation on project creation (background job)

**Frontend:**
- `VideoEditorMode` full-screen overlay, routed from Lightbox
- `nleStore` with timeline state, undo/redo
- `PreviewArea` with canvas fed from WebSocket frame stream
- Minimal `Timeline` with single track, single clip, playhead
- `InspectorPanel` with clip properties (trim, speed, volume, fade)
- Effects panel (reuse `AdjustPanel` + `HslPanel` from photo editor)
- Export dialog (resolution, fps, quality, save as/overwrite)
- Lightbox routes videos to NLE instead of photo editor

### Phase 2 — Multi-Clip Assembly (3-4 weeks)
Multiple clips on track, drag reorder, trim handles, split at playhead, zoom.

- "Add Clip" button → media browser (list of user's videos from Prism library)
- `ClipElement` with draggable + resizable trim handles
- Split at playhead (`S` key), delete clip (`Delete`), ripple delete
- Clip reordering via drag within track
- Timeline zoom (buttons + scroll wheel)
- Time ruler with tick marks
- Keyboard shortcuts: Space, arrows, S, Delete
- `POST /nle/clips/thumbnail-strip` for timeline filmstrip

### Phase 3 — Multi-Track + Compositing (3-4 weeks)
Multiple video/audio tracks, transform per clip, more transition types.

- Multiple video tracks (overlays, picture-in-picture, B-roll)
- Multiple audio tracks with independent gain/mute/solo
- Transform controls per clip (position/scale/rotate)
- More transition types (wipe, slide, dissolve variants)
- Track add/remove/reorder
- Solo button per track

### Phase 4 — Keyframes + Speed (3-4 weeks)
Generic keyframe editor, speed ramps, reverse playback, freeze-frame.

- `KeyframeEditor` component (generic curve editor for any parameter)
- Keyframeable: opacity, transform, volume, filter intensity
- Speed ramps: constant speed + eased curves
- Reverse playback
- Freeze-frame insertion
- Keyframe interpolation modes (linear, ease-in, ease-out, bezier)

### Phase 5 — Color + Audio + Polish (3-4 weeks)
Color grading, audio ducking/normalization, export presets, project templates.

- Color grading via MLT's color correction filters (exposure, contrast, saturation, temperature, LUT support)
- Audio ducking (auto-lower background when voice is present)
- Audio normalization
- Export presets (Instagram Reel, YouTube 1080p, TikTok, custom)
- Project templates
- Before/after comparison toggle
- Auto-save

## Key Technical Decisions

1. **MLT as rendering engine** — same code path for preview and export, no drift
2. **Project JSON as source of truth** — independent of MLT, enables future interop
3. **Proxy-based preview** — low-res proxies for scrubbing, full-res for export, user never sees it
4. **DOM-based timeline** — sufficient for <50 clips, easier to debug than canvas
5. **Reuse photo editor panels** — `AdjustPanel`, `HslPanel`, `DetailPanel` work as-is
6. **Non-destructive** — original files never modified, project data stores all edits
7. **Export user choice** — save as new file or overwrite original
8. **WebSocket for frame streaming** — lower latency than REST polling for preview playback

---

## UI Design Principles ("Simpler Than Premiere")

- **One timeline view, not a dozen panels** — Premiere's complexity is 80% panel management
- **Smart defaults over dialogs** — drag clip onto clip → auto crossfade, advanced panel available but never required
- **Contextual tools** — show color grading only when a clip with that effect is selected
- **Magnetic timeline + snapping on by default** — this is why CapCut feels easier than Premiere
- **No project format complexity exposed** — no "media offline", no proxy management UI, no codec selection unless exporting
