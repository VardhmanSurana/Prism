# PaddleOCR-VL Integration Plan

## Summary
Add OCR text extraction to the photo processing pipeline as **Stage 4**, running PaddleOCR-VL GGUF models via the existing `llama-server` infrastructure on port 9092. Extracted text is stored in a new `ocr_text` column and made searchable via FTS5.

### Step 2: Photo Model — `models.py`

### Step 3: Dynamic Migration + FTS5 — `main.py`

**3a.** Add `ocr_text` column migration after existing ALTER TABLE block (after line 69):
```python
if "ocr_text" not in columns:
    await conn.execute(text("ALTER TABLE photos ADD COLUMN ocr_text TEXT"))
```

**3b.** FTS5 migration — FTS5 virtual tables cannot be ALTERed. Check if `ocr_text` exists in FTS5 columns; if not, DROP and recreate with all triggers:
- Check via `PRAGMA table_info(photos_fts)`
- If `ocr_text` missing: DROP table, DROP triggers, CREATE new table with `ocr_text`, recreate all 3 triggers (INSERT/UPDATE/DELETE) including `ocr_text`, re-populate from photos

**Updated FTS5 triggers** must include `ocr_text`:
- INSERT: add `new.ocr_text` to columns list
- UPDATE: add `ocr_text = new.ocr_text` to SET clause
- DELETE: unchanged (deletes by photo_id)

### Step 4: AIOrchestrator — `ai_orchestrator.py`

elif mode == 'ocr':

### Step 5: OCR Service Module — `services/ocr/`

**`__init__.py`**: Export `OCRManager`, `extract_ocr_text`

**`ocr_manager.py`**: Follows `VisionManager` pattern exactly:
- `OCRManager.get_ocr()` → starts llama-server in 'ocr' mode, returns `query_server` callable
- `OCRManager.query_server(image_path)` → encodes image to base64, sends multimodal chat completion to `/v1/chat/completions` with prompt "Extract all visible text from this image. Return only the extracted text, preserving line breaks."
- `OCRManager.unload()` → calls `AIOrchestrator.stop_server()`

**`ocr_extract.py`**: Sync wrapper `extract_ocr_text(image_path: str) -> str | None` for use with `asyncio.to_thread`

### Step 6: Processing Queue Stage 4 — `processing_queue.py`

**6a.** Add to results init dict (~line 241):
```python
"ocr_text": None,
"stage4_success": True,
```

**6b.** Add Stage 4 block after Stage 3 DB save (after line 436):
- Gate on `settings.ENABLE_AI_OCR`
- Load OCR server via `OCRManager.get_ocr()`
- Process batch: `await asyncio.to_thread(extract_ocr_text, path)`
- Unload via `OCRManager.unload()`

**6c.** Add OCR text DB save block after Stage 4:
- Save `photo.ocr_text` for each photo with results

**6d.** Update job status logic (line ~446):
- Add `stage4_needed` and `s4_ok` checks
- OCR is optional — `success = s2_ok and s3_ok` (stage4 failure doesn't fail job)

**6e.** Update `_broadcast_status`:
- Add `ocr_processed` count query (photos with `ocr_text IS NOT NULL`)
- Add `"ocr"` key to status_data

### Step 7: Search Tools — `search_tools.py`

Update `search_ocr` (line 270) to:
1. Try FTS5 MATCH first (since `ocr_text` is now in `photos_fts`)
2. Fall back to ILIKE on `Photo.ocr_text` (alongside existing filename/caption/ai_summary)

### Step 8: On-Demand API Endpoint — `api/photos/ocr.py`

New endpoint: `POST /api/v1/photos/{photo_id}/ocr`
- Guard: check `ENABLE_AI_OCR`
- Load photo, check not locked
- Call `extract_ocr_text(photo.path)`
- Save to `photo.ocr_text`, commit
- Return `{ photo_id, ocr_text }`
- Unload OCR server in `finally` block

### Step 9: Wire Up — `api/photos/__init__.py` + `metadata.py` + `main.py`

- `__init__.py`: Import and export `ocr_router`
- `metadata.py`: Add `"ocr_text": photo.ocr_text` to metadata response dict
- `main.py`: Register OCR router with `dependencies=[Depends(verify_api_key)]`
